# 项目愿景（PROJECT_VISION）

## 为什么重启新仓库

旧有声书相关仓库结构混乱、文档与代码脱节、难以让 AI Agent 按轮次持续推进。新仓库 `audiobook-cleaner-lab` 从零建立：

- 清晰的 Stage/Round 路线图
- 可机器读取的治理协议
- 非破坏式 JSON 切点设计
- 模块化 Adapter（ASR / LLM 可替换）

目标不是「一轮做完所有功能」，而是**长期可维护的个人生产工具**。

## 为什么使用单仓库

有声书清洗流水线高度耦合：原文、ASR、对齐、机切、Review、导出、反馈共享同一套 JSON schema 与 data 目录结构。单仓库便于：

- 统一版本化 schema 与文档
- Agent 按 round 文件跨模块推进
- 本地 `data/` 工作区与代码同机管理

内部仍按 `apps/` + `packages/` 模块化，避免 spaghetti。

## 为什么要结合原文文本

仅有 ASR 无法区分：

- 正文 vs 录音时的废话
- 读错后废弃的片段 vs 最终正确朗读
- 重读前的错误尝试

正确原文是**语义 ground truth**。ASR 提供**时间锚点**。二者结合才能定位「多读」「漏读」「偏离原文」的片段。

## 为什么要结合 ASR

原文没有天然时间戳。ASR segment 的 `start`/`end` 是将文本判断落到音频切点的桥梁。没有 ASR（或等价的手动 transcript），无法自动生成切点区间。

## 为什么要使用大模型

规则与正则无法覆盖所有口误、重读、自言自语场景。大模型擅长在**对齐结果 + 上下文**下判断某 ASR 片段是否属于正文、是否为废话或废弃重读。

但 LLM 的职责边界严格限定为**文本层结构化建议**，见 `LLM_CUT_DECISION_PROTOCOL.md`。

## 为什么必须有人工校正页面

- ASR 有错字、时间戳漂移
- LLM 可能误判正文为废话
- padding 需听感微调
- 个人有声书质量要求高于全自动

理想流程：**机切生成建议 → 网页试听确认 → 保存 cut_plan → FFmpeg 导出**。

## 为什么不追求一开始全自动

录音场景多样（咳嗽、试读、对自己说话、章节间说明）。全自动误删正文的代价远高于多花时间 Review。第一阶段目标是**可靠的半自动流水线**，反馈闭环逐步提升机切准确率。

## 最终希望形成的个人生产工具

一台本地运行的「有声书后期工作站」：

1. 导入一书多章音频与原文
2. 批量 ASR → 对齐 → 机切建议
3. 网页 Review + 波形微调切点
4. 一键 FFmpeg 导出干净章节
5. 修正记录反哺规则与 prompt

适合个人创作者，非 SaaS，非 DAW 替代品。

## 当前默认假设

- 语言：中文有声书为主
- 用户：单人本地使用
- 部署：本地 Python + 浏览器，无云依赖（API 可选）
